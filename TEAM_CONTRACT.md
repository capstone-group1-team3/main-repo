## 3. Conflict Plan

### 3.1 Process for Resolving Disagreement

When disagreement happens, we will handle it directly and respectfully. First, we name the specific disagreement, such as scope, architecture, evaluation, UI, or timeline. Then each teammate briefly explains their view and concern. The team will look for a consensus decision that protects the MVP and aligns with the rubric. If consensus is not reachable quickly, we will choose the smallest workable compromise and document the decision in the board card, README notes, or an architecture decision note.

### 3.2 If One Teammate Is Dominating or Steamrolling

- **First step:** A teammate will name the concern kindly and directly during the team sync or in a private message: "I think we need to slow down and make sure everyone understands and agrees before changing this part."
- **Team correction:** We will pause the decision, restate the options, and give every teammate a chance to speak.
- **Escalation:** If the pattern continues, we will ask the Support Instructor for facilitation. If still unresolved, we will escalate to the Lead Instructor.

### 3.3 If One Teammate Is Under-Contributing or Missing Check-ins

- **First step:** One teammate will check in privately and respectfully to ask whether the person is blocked, confused, unavailable, or overloaded.
- **Support step:** The team will help reduce ambiguity by assigning a smaller, clear task with acceptance criteria and a pairing option.
- **Escalation threshold:** If the teammate misses a required sync or does not respond to important messages for 24 hours without notice, the team will flag the issue in the team channel and agree on a recovery plan.
- **Escalation:** If there is still no response or the timeline is affected, we will escalate to the Support Instructor first, then the Lead Instructor if needed.

### 3.4 Handling Mismatched Skill Levels

We expect different teammates to be stronger in different technical areas. More-experienced teammates will coach by explaining, pairing, reviewing, and leaving notes, not by taking over the task. Less-experienced teammates will ask for help early instead of disappearing into a rabbit hole. If someone is stuck for more than 60 minutes on a technical blocker, they must ask for help from the team.

### 3.5 Escalation Path

If the team cannot resolve a conflict internally, we will escalate in this order:

1. **Support Instructor** for facilitation and team-process support.
2. **Lead Instructor** for final decision if needed.

All teammates agree to this escalation order.

Every graded part of the working project lives in `main-repo`, including code, tests, README, executive briefing, architecture diagram, runbook, Dataset Card or Model Card, demo scripts, and final documentation.

### 6.2 Branch Strategy

- **`main`:** Production-ready branch. All PRs land here only after review.
- **`dev`:** Optional staging branch. We will skip it unless integration becomes complex enough to require it.
- **Feature branches:** One task per branch, using one of these formats:
  - `feature/<short-task-name>`
  - `fix/<bug-name>`
  - `docs/<doc-name>`
  - `eval/<eval-task>`
  - `<github-handle>/<short-task-name>`

Examples:

- `feature/rag-pipeline`
- `feature/langgraph-router`
- `fix/docker-compose`
- `docs/update-readme`
- `eval/heldout-set`

Branches should be deleted after merge.

### 6.3 PR Review Workflow

Every PR to `main` requires **two teammate approvals** before merging. The PR author does not approve their own PR. Two other teammates approve, and after the second approval, either a reviewer or the author may click merge only if the branch protection rules allow it and all checks are satisfied.

PR title format:

```text
<type>/<slug>: <short imperative summary>
```

Example:

```text
feature/eval-harness: add held-out set loader
```

Every PR body must include:

```md
## What changed?
Explain the change.

## Why?
Explain why this task matters.

## Test evidence
Paste pytest output, screenshots, or manual test notes.

## Notes for reviewers
Mention what reviewers should focus on.
```

Additional PR rules:

- Keep PRs small and reviewable.
- Link the PR to the relevant GitHub Project card.
- If new commits are pushed after approval, stale approvals should be dismissed and the PR must be reviewed again.
- No one bypasses branch protection.
- No direct pushes to `main`.

### 6.4 Merge Cadence

We will merge to `main` daily when possible, especially after team syncs or after a blocker is resolved. A teammate marks a PR ready for review in the Slack channel and tags the two reviewers. If a PR is urgent because it unblocks another teammate, the author should explain the urgency clearly.

### 6.5 GitHub Project Board — How Our Team Runs It

Columns:

- `Backlog`
- `In Progress`
- `In Review`
- `Done`

Rules:

- Every task is a card.
- Every card has an owner.
- No unassigned card stays in `In Progress`.
- When work starts, the card moves to `In Progress`.
- When a PR opens, the card moves to `In Review`.
- When the PR merges, the card moves to `Done`.
- If a card stays in `In Progress` for more than 24 hours without a note, someone asks about it at standup.

Daily standup rhythm:

- Each teammate answers:
  1. What did I finish yesterday?
  2. What will I finish today?
  3. What is blocking me?
- Maximum length: 15 minutes.
